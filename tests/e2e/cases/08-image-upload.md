# 08-image-upload: 改造场景截图上传 + AI 描述

**规则**（来自 需求 v3 §3.2、§4.4 + `prompts.ts` "反常规对话场景"节）：
- 用户可在非 stage 1 任意阶段上传截图
- AI 先"描述我看到了什么"再对齐，不假装看懂

## 前置

- 干净加载
- 准备一张可用的测试图片：可用任意截图。tester 从 `tests/e2e/fixtures/sample-screenshot.png` 读（若不存在，从 dev server 的任意 404 页面截一张临时图也行，但至少保证能触发 upload）

## 步骤

1. 用自然语言描述一段：`现在我们的后台员工列表页很难用，想改`
2. 等 AI 回复
3. 在输入框旁找到**上传图片**入口（可能是回形针图标、相机图标或 `<input type="file">`）
4. 上传一张图片（fixture 或临时占位图）
5. 在同一条消息里补充文字描述（若 UI 支持）：`这是现在的样子，字段太多一屏放不下`
6. 发送
7. 等 AI 回复完成
8. 读取 AI 回复气泡文字

## 期望

- 步骤 3-4：上传图片的 UI 入口可点且交互后图片缩略图或文件名在输入区出现
- 步骤 6：用户消息气泡出现，含刚才的图片（`<img>` 元素可见，src 含 blob: / data: / 图片 URL）
- 步骤 8：AI 回复气泡文字 ≥ 40 字符，**且至少满足下列一条**描述行为：
  - 文字包含"看到"、"注意到"、"观察到"、"截图里"、"图里"、"这个页面"之一（AI 在描述所见）
  - 文字包含具体的视觉元素词（如"表格"、"列表"、"字段"、"按钮"、"标题"之一）
- 不应出现"图片上传失败"、"无法识别图片"类错误提示

## 通过标准

图片能上传、AI 能接收并产出一段描述其所见的回复。

## 注意

本 case 可能因图片内容差异导致 AI 回复风格不同，但"**AI 能收到并语义上回应图片内容**"这一基本能力必须稳定。若 fixture 缺失，tester 可用浏览器内 canvas 现场生成一张带"员工列表 姓名 工号 部门"中文字的 200x200 PNG 临时写到 `tests/e2e/artifacts/` 再上传。
